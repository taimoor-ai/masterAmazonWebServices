Phase 01 / Foundation
Cloud Fundamentals That Actually Matter
Don't just memorize AWS services. Understand WHY cloud exists: elasticity, pay-as-you-go, shared responsibility. Your Node.js background is a superpower here — deploy real apps from day 1.
Weeks 1–8
AWS Services to Learn
IAM EC2 S3 VPC Route 53 CloudFront RDS (PostgreSQL) Lambda (intro) CloudWatch AWS CLI Elastic Load Balancer Auto Scaling Groups
Core Concepts — Not Just Tools
  • Shared Responsibility ModelAWS secures the cloud; you secure what's IN the cloud. Every security mistake traces back to misunderstanding this.
  • Networking fundamentalsCIDR blocks, subnets, routing tables, security groups vs NACLs. You can't troubleshoot without this.
  • IAM Principle of Least PrivilegeRoles over users. Service accounts. Never use root. Understand policies as JSON.
  • Stateless vs Stateful designWhy EC2 needs an ELB in front. Why S3 is infinitely scalable but your DB isn't.
  • Availability Zones vs RegionsMulti-AZ is for availability; Multi-Region is for disaster recovery. Know the cost difference.
DevOps Tools — Phase 1 Intro
  • Docker (containers)Containerize your Node.js apps. Understand images, layers, volumes. Build a Dockerfile before touching ECS.
  • Linux CLI basicsSSH into EC2, manage permissions, read logs, use systemd. You'll use this daily.
  • Git + GitHubBranching strategies (GitFlow). This is version control for your infra later.
  • Bash scriptingAutomate repetitive tasks. User data scripts for EC2 bootstrap. Basic loops and conditionals.
Real-World Projects
P1-A
Deploy your Node.js API to EC2 — Production-style
Not just "run npm start on EC2". Do it properly: Nginx reverse proxy, SSL via Let's Encrypt, systemd process management, custom VPC with public/private subnets.
What you'll build
Custom VPC + subnets EC2 behind Nginx RDS in private subnet S3 for file uploads CloudFront CDN Route53 custom domain
Hands-on tasks
  • Create a VPC with 2 public + 2 private subnets across 2 AZs
  • Launch EC2, SSH in, install Node.js via NVM — zero tutorial shortcuts
  • Configure Nginx as reverse proxy to Node.js port 3000
  • Set up RDS PostgreSQL in private subnet, connect from EC2 via security group only
  • Add S3 bucket for user uploads (server-side using SDK, NOT public bucket)
  • Enable CloudWatch monitoring + create CPU alarm that sends email
  • Add Auto Scaling Group with min=1, max=3, scale on CPU > 70%
P1-B
Containerize Everything with Docker
Take the same Node.js API and fully containerize it. This bridges phase 1 to phase 2 (ECS/EKS).
  • Write multi-stage Dockerfile (build stage → production image, reduces image size by 70%+)
  • Use docker-compose for local: api + postgres + redis
  • Push image to Amazon ECR (Elastic Container Registry)
  • Run your container on EC2 (pre-ECS warmup)
Common Mistakes
⚠
Leaving resources running (destroys your budget)

Set AWS Budgets to $20/month with email alert on day 1. Stop EC2 instances when done. Delete RDS snapshots. This is the #1 mistake of beginners.

⚠
Putting everything in a public subnet

Your database should NEVER be in a public subnet. Private subnet + security group allowing only your EC2 is the correct pattern.

✓
Use AWS Free Tier strategically

t2.micro/t3.micro EC2, RDS db.t3.micro, 5GB S3 — all free for 12 months. Learn cost optimization from day 1.

Phase 02 / Intermediate
Production Architecture + DevOps Pipeline
This is where most engineers plateau. The difference between a junior and mid-level cloud engineer is automated deployments, infrastructure as code, and serverless design patterns.
Weeks 9–20
AWS Services to Learn
ECS Fargate API Gateway Lambda (deep) SQS SNS DynamoDB ElastiCache (Redis) CodePipeline CodeBuild ECR Secrets Manager Systems Manager (SSM) EventBridge Step Functions
Core Concepts
  • Infrastructure as Code (IaC) with TerraformStop clicking the console. Every resource you create must be reproducible. Learn HCL: providers, resources, variables, outputs, state files, remote state on S3.
  • Event-Driven ArchitectureSQS decouples producers from consumers. SNS fans out to multiple subscribers. EventBridge is the glue for async workflows. This is the real-world pattern for high-throughput systems.
  • CI/CD Pipeline DesignSource → Build → Test → Deploy. Blue/green deployments. Canary releases. Zero-downtime deploys. These aren't optional for production systems.
  • Serverless design patternsCold starts, concurrency limits, execution time limits, idempotency. Lambda + API Gateway isn't just "run function" — it's an architectural decision with tradeoffs.
  • Secret managementNEVER hardcode secrets. Secrets Manager vs SSM Parameter Store. IAM roles for service-to-service auth. Envelope encryption.
DevOps Tools — Phase 2 Deep Dive
  • Terraform (core skill)Modules, workspaces, state locking with DynamoDB. Write Terraform for everything from Phase 1. Never provision manually again.
  • GitHub Actions CI/CDBuild pipelines: lint → unit test → build Docker image → push to ECR → deploy to ECS. Learn OIDC authentication with AWS (no long-lived access keys in CI).
  • AWS CDK (optional but valuable)If you prefer TypeScript, CDK generates CloudFormation. Great bridge from dev → ops thinking.
  • Monitoring: CloudWatch + GrafanaCustom metrics, log groups, dashboards. Trace a request end-to-end with X-Ray. Alarm on business metrics, not just CPU.
Real-World Projects
P2-A
Fully-Automated E-Commerce Backend
Microservices on ECS Fargate, with a proper CI/CD pipeline. This is the project that separates you from 80% of CS graduates.
Architecture
3 microservices on ECS Fargate API Gateway + Lambda auth SQS order queue DynamoDB for cart Redis for sessions SNS email notifications
CI/CD Pipeline (hands-on)
  • GitHub Actions: PR triggers lint + unit tests (jest); merge to main triggers deployment
  • Build Docker image, tag with git SHA, push to ECR
  • Use OIDC (not access keys) for GitHub → AWS auth
  • Deploy to ECS with rolling update strategy
  • Write ALL infrastructure in Terraform (VPC, ECS cluster, task definitions, IAM)
  • Store Terraform state in S3 + DynamoDB lock table
  • Implement health check endpoint, ALB routes only to healthy tasks
P2-B
Serverless Image Processing Pipeline
User uploads photo → S3 → Lambda triggered → resize to 3 sizes → store back to S3 → notify via SQS. A classic real-world event-driven pattern.
  • S3 bucket triggers Lambda on object creation (not polling — event-driven)
  • Lambda uses Sharp library (deployed as Lambda Layer)
  • Dead Letter Queue (SQS) for failed processing jobs
  • X-Ray tracing to measure cold start vs warm performance
  • Step Functions for multi-step approval workflow (bonus)
Common Mistakes
⚠
Treating Terraform as "just a tool" — not understanding state

State drift kills production systems. Always use remote state. Never run terraform apply locally in a team. Use workspaces for dev/staging/prod environments.

⚠
Lambda anti-patterns: putting everything in one function

Each Lambda should have a single responsibility. Don't put your entire Express app in one Lambda — that defeats serverless. Use Lambda for event handlers, not monoliths.

◆
DynamoDB access patterns MUST be designed upfront

Unlike MongoDB, you design your DynamoDB schema around your queries, not the other way around. Model your single-table design before writing any code.

Phase 03 / Advanced
System Design + Kubernetes at Scale
This is where you start thinking like a staff engineer. Reliability engineering, cost optimization, Kubernetes orchestration, and distributed systems design. Your FYP-winning project lives here.
Weeks 21–32
AWS Services to Learn
EKS (Kubernetes) Aurora Serverless v2 OpenSearch Kinesis Data Streams WAF + Shield CloudTrail Config Cost Explorer + Budgets Service Mesh (App Mesh) GuardDuty Macie Organizations + SCPs
Core Concepts
  • Kubernetes internalsDon't just run kubectl. Understand: Pods, Deployments, Services, Ingress, ConfigMaps, Secrets, RBAC, HPA (Horizontal Pod Autoscaler), resource limits/requests. EKS adds: IAM for service accounts (IRSA), managed node groups, Fargate profiles.
  • Distributed systems patternsCircuit breaker, retry with exponential backoff, idempotency keys, saga pattern for distributed transactions. These prevent cascading failures at scale.
  • Observability (not just monitoring)The 3 pillars: Logs, Metrics, Traces. Implement OpenTelemetry. Correlate a slow API request back to a specific DB query across 5 microservices.
  • Cloud security postureGuardDuty for threat detection. AWS Config for compliance drift. SCPs to prevent dangerous actions across an org. Zero-trust networking.
  • FinOps / Cost engineeringReserved instances vs Savings Plans vs Spot. Right-sizing with Compute Optimizer. Tagging strategy for cost allocation. This is what CTOs ask about.
DevOps Tools — Phase 3
  • Helm (Kubernetes package manager)Deploy applications with Helm charts. Write your own chart. Understand values.yaml, templates, and chart versioning.
  • ArgoCD (GitOps)GitOps = your Git repo IS the source of truth for cluster state. ArgoCD watches your repo and auto-syncs. This is how serious teams do K8s deployments.
  • Prometheus + Grafana on EKSDeploy kube-prometheus-stack via Helm. Build dashboards for pod CPU/memory, HTTP error rates, custom business metrics.
  • Terraform advancedModules as reusable components. Terragrunt for DRY configs. Terraform Cloud for team workflows. Drift detection.
  • chaos engineering introAWS Fault Injection Simulator. Kill a pod randomly. Does your system recover? If not, you need better resilience patterns.
FYP / Job-Winning Project
🏆 Real-Time Multi-Tenant SaaS Analytics Platform
A production-grade, multi-tenant analytics dashboard that ingests live event streams, processes them at scale, stores results in a time-series DB, and displays them on a React frontend. This hits every hot topic: Kubernetes, real-time streaming, multi-tenancy, security isolation, and cost optimization.
Data Ingestion Layer
API Gateway + Lambda Kinesis Data Streams Kinesis Firehose → S3
Processing Layer
EKS microservices Kafka (MSK) for event bus Lambda for aggregations
Storage Layer
DynamoDB (per-tenant) Aurora Serverless (reports) S3 + Athena (raw data lake)
Frontend + Security
React dashboard (WebSockets) Cognito + tenant isolation WAF + rate limiting
Why this wins FYP / job interviews
  • Multi-tenancy is a hard engineering problemShows you understand data isolation, tenant-specific rate limiting, and SaaS architecture patterns that companies pay senior engineers to design.
  • Real-time streaming is in-demandKinesis + Kafka experience makes you competitive for data engineering and backend roles, not just cloud ops.
  • Full-stack cloud storyYou built the infra (Terraform), the API (Node.js), the processing (Lambda/EKS), the storage (multi-DB), and the frontend (React). Zero gaps in your knowledge.
  • Demonstrable at scaleLoad test with k6 to show it handles 10,000 events/second. Put the numbers in your CV and FYP report.
⚠
EKS is expensive — plan accordingly

An EKS cluster costs ~$72/month for the control plane alone, before nodes. For learning: use eksctl with spot instances, shut down non-working hours, or use k3s/Minikube locally and only use EKS for final demos.

⚠
Don't learn Kubernetes before understanding containers deeply

EKS is complex. If you don't understand Docker networking, volume mounts, and multi-container pods yet, ECS Fargate is the correct intermediate step. Rushing to K8s without this foundation creates dangerous half-knowledge.

Phase 04 / Elite
Staff-Level Cloud Architecture
The top 10% doesn't just deploy services. They design for failure, optimize for cost at scale, define org-wide security posture, and can lead a team. This phase is about thinking, not just doing.
Weeks 33–48
Advanced Architecture Patterns
  • Well-Architected Framework (deeply)6 pillars: Operational Excellence, Security, Reliability, Performance Efficiency, Cost Optimization, Sustainability. Run Well-Architected Reviews on your own projects and fix the gaps you find.
  • Multi-region active-activeRoute 53 latency-based routing + health checks. DynamoDB Global Tables. Aurora Global Database. RTO/RPO planning. This is disaster recovery architecture, not just backup.
  • Service mesh & zero-trustIstio/App Mesh for mTLS between services. Every service-to-service call authenticated and encrypted. No implicit trust inside the VPC.
  • Platform engineeringBuild internal developer platforms. Golden paths. Backstage.io. Your job isn't to be the only one who knows AWS — it's to make your team productive on AWS.
Elite Engineering Mindset
  • Reliability engineering (SRE)Define SLIs, SLOs, error budgets. Runbooks for every alert. Blameless post-mortems. On-call rotations. This is the operational discipline that separates good from elite.
  • Cost at scaleSpot instance strategies with capacity-optimized allocation. Graviton3 (ARM) for 40% cost savings. Intelligent-Tiering for S3. Schedule-based scaling. Savings Plans analysis.
  • Writing Architecture Decision Records (ADRs)Document WHY you made an architectural decision, not just what. Future-you and teammates will thank you.
  • Contribute to open sourceTerraform providers, CDK constructs, Helm charts. This is tangible proof of expertise that no certification can match.
Elite-Level Projects
P4-A
Multi-Region Disaster Recovery System
Design a system that achieves RTO < 5 minutes and RPO < 1 minute across two AWS regions. Document the architecture decision and tradeoffs.
  • Aurora Global Database with automated failover
  • DynamoDB Global Tables (multi-master)
  • Route 53 Application Recovery Controller
  • Simulate regional failure and measure actual RTO vs target
  • Write a runbook and post-mortem template
P4-B
Internal Developer Platform (IDP)
Build a self-service portal where developers can provision environments with pre-approved Terraform templates. This is what platform engineering looks like.
  • Backstage.io deployed on EKS with AWS integrations
  • Service catalog of approved infrastructure templates
  • Automated cost tagging + chargeback reporting
  • Security guardrails: SCPs prevent deviation from approved patterns
◆
The biggest mistake at elite level: staying technical only

Top 10% engineers communicate architecture to non-engineers, write clear documentation, mentor others, and understand business impact (cost, time-to-market, risk). Start writing blog posts or a technical portfolio now — it compounds.
